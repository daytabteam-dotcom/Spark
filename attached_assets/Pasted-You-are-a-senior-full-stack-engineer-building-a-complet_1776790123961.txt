# BACKEND API ROUTES

## Auth
POST /auth/google — receive Google OAuth token, verify, create/find user, return JWT
POST /auth/refresh — refresh JWT
GET  /auth/me — return current user profile

## Users
PATCH /users/me — update name, selected_character_id, difficulty_tags
GET   /users/me/xp — return xp total + unlocked characters

## Tasks
GET    /tasks — list all tasks for user (filter by status)
POST   /tasks — create task
PATCH  /tasks/:id — update task (status, scheduled_for, etc.)
DELETE /tasks/:id — delete task
POST   /tasks/extract — given raw user text, use AI to extract structured tasks, return preview (do NOT save yet)
POST   /tasks/confirm-extract — save extracted tasks after user confirms

## Chat
POST /chat/stream — stream AI response (SSE)
  Body: { character_id, message, session_id, recent_messages[] }
  Server builds context: character definition + user profile + session summary + last 6 messages
  Returns SSE stream with text chunks + optional UI_BLOCK metadata

## Sessions
POST  /sessions — start new session
PATCH /sessions/:id — update (end session, attach summary)
GET   /sessions/:id — get session
POST  /sessions/:id/summarize — trigger AI summary of session

## XP
POST /xp/award — award XP for an event (task_completed, focus_session, streak, return_after_distraction)
  Automatically checks unlock thresholds and unlocks characters

## Scheduling
POST /schedule/delay — schedule a task for later (10min | 30min | later_today | tomorrow)
  Creates a Bull job that fires a push notification at the scheduled time

## Notifications
POST /notifications/register — register device push token

# AI SERVICE

Build an `ai-service.ts` module with:

## buildContext(userId, characterId, sessionId, recentMessages)
Fetches:
- Character personality_prompt from seed data
- User name + difficulty_tags + selected_character
- Session summary (if exists)
- Last 6 messages only

Returns a messages array for the Anthropic API:
- system: "[character personality] + [user context] + [session summary] + [behavioral instructions]"
- messages: last 6 from recentMessages

## streamChatResponse(context, onChunk, onUIBlock)
Calls Anthropic API with streaming enabled.
Parses response for optional UI_BLOCK markers in this format:
[UI_BLOCK:{"type":"task_card","data":{...}}]
[UI_BLOCK:{"type":"quick_actions","options":["Start now","In 10 min","Tomorrow"]}]
[UI_BLOCK:{"type":"checklist","items":["item1","item2"]}]
[UI_BLOCK:{"type":"schedule_selector"}]
[UI_BLOCK:{"type":"resistance_options","options":["It feels too big","I don't know where to start","I just don't want to"]}]
[UI_BLOCK:{"type":"focus_controls"}]
[UI_BLOCK:{"type":"session_summary","tasks_completed":[],"tasks_partial":[],"xp_earned":0}]

Strips UI_BLOCK markers from visible text, calls onUIBlock callback with parsed block data.

## extractTasks(rawText)
Sends raw user brain-dump text to Anthropic API.
System prompt instructs extraction of tasks in this JSON format:
{
  "tasks": [
    {
      "title": string,
      "duration_minutes": number | null,
      "resistance_level": "low" | "medium" | "high",
      "task_type": string | null
    }
  ]
}
Returns parsed task array.

## generateSessionSummary(sessionId, completedTasks, partialTasks, durationMinutes)
Returns a short 2-3 sentence session summary in the character's voice.

# CHARACTER BEHAVIORAL INSTRUCTIONS

Each character must have a personality_prompt. Include these in seed data and use them in the AI system prompt.

Also include these global behavioral rules in every system prompt:

- You are an AI companion inside an ADHD productivity app. Your job is to help the user identify tasks, start tasks, overcome procrastination, stay focused, and complete tasks.
- Keep ALL messages short: 1-3 sentences max. ADHD users do not read long messages.
- Always include at least one UI_BLOCK per response. Never send a text-only response.
- Minimize typing. Offer buttons and options wherever possible.
- One action at a time. Never overwhelm.
- Detect the user's behavioral state: idle | resistance | active | distraction | return | completion | overwhelm. Respond to their state, not just their words.
- When user seems stuck, offer resistance_options UI block.
- When user completes something, celebrate briefly then move forward. Do not over-praise.
- Never moralize. Never lecture. Never say "you should".
- Always speak in character voice and personality.

# XP SYSTEM

XP is awarded on these events:
- task_completed → +20 XP
- focus_session_completed (>10 min) → +15 XP
- return_after_distraction → +10 XP
- daily_streak → +25 XP

On each XP award:
1. Update users.xp in database
2. Insert into xp_events table
3. Check if new XP total crosses any unlock threshold
4. If yes: mark character as unlocked (store unlocked_characters array on user or join table)
5. Return { newXP, unlockedCharacter } in response so client can show unlock animation

# SCHEDULING SYSTEM

When user delays a task:

POST /schedule/delay
Body: { taskId, delay: "10min" | "30min" | "later_today" | "tomorrow" }

1. Calculate scheduled_for timestamp
2. Update task.scheduled_for and task.status = "scheduled"
3. Create a Bull delayed job
4. When job fires: send push notification in character voice
   Example (Entropy Fox): "Hey. That thing you were going to do? Still there. Waiting. Your call."

If task is missed (scheduled_for passed, status still "scheduled"):
- Bull job fires a second notification
- Message offers: reschedule | make it smaller | skip it

# SESSION SYSTEM

Sessions track a single focused work period.

Start session: POST /sessions → returns session_id
During session: all chat messages tagged with session_id
End session: triggered by inactivity (Bull job, 10 min no messages) or explicit end
On end:
1. Call generateSessionSummary
2. Store summary in sessions.summary
3. Display SessionSummaryCard in chat

Do NOT store full chat history. Store only the session summary.
Use Redis to cache the last 20 messages per session (keyed by session_id).
Evict when session ends.

# BEHAVIOR STATES

Detect and log these states via behavior_log table:

- idle: no messages for >5 minutes
- resistance: user says they can't start, keeps delaying
- active: user is sending messages, tasks in progress
- distraction: user opened app mid-task but hasn't responded
- return: user comes back after idle/distraction
- completion: task marked completed
- overwhelm: user mentions too many tasks, feeling stuck

Map each state to character response strategy:
- idle → gentle nudge in character voice
- resistance → offer resistance_options block
- active → affirm and continue
- distraction → acknowledge return without guilt
- return → celebrate return, re-orient to task
- completion → brief celebration, ask about next task
- overwhelm → acknowledge, offer to pick ONE task together
